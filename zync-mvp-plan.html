Feature Nuevo: Venta de Entradas

🎟 Flujo de venta de entradas para eventos

El negocio define un precio de entrada para cada evento. El usuario lo compra desde la app, el sistema genera un ticket con QR único, y el negocio escanea el QR en la puerta para validar el acceso.

1
Business crea evento con precio
→
2
User ve evento y compra entrada
→
3
Se descuenta del wallet
→
4
App genera ticket + QR
→
5
Business escanea QR en puerta
→
6
Acceso validado ✅
👤
ROL: USER
Asistente — Descubre eventos, compra entradas, pide al bar, solicita canciones al DJ
Backend Endpoints a implementar / completar
B-U1 QR de entrada al establecimiento
POST/qr/scan
Recibe payload del QR (venueId + eventId). Devuelve { venue, event, products[] }. Registra al usuario como "dentro" del establecimiento.
GET/venues/:venueId/active-event
Devuelve el evento activo del venue si existe, sin que el usuario lo busque manualmente.
B-U2 Menú / Productos accesibles para usuario
GET/venues/:venueId/products?category=bebidas
Endpoint ya existe solo para Business. Abrir para JWT de usuario normal. Añadir campo category a Product y permitir filtro.
  • Producto debe incluir: name, description, price, category, imageUrl, available
B-U3 Order status en tiempo real
GET/orders/:orderId/status
Para polling si el socket no está conectado.
  • Socket event: order:status_update → { orderId, status, estimatedMinutes? }
  • El negocio cambia el estado → el backend emite el evento al usuario
  • Estados: pending → confirmed → ready → delivered
B-U4 Wallet / Top-up
POST/wallet/topup
Recibe { amount, paymentMethodToken? }. Actualiza user.balance. MVP puede ser mock sin pasarela real.
GET/wallet/balance
Devuelve { balance, zyncPoints } para refrescar sin recargar el perfil completo.
B-U5 Estadísticas de perfil (tier, gasto)
GET/users/me/stats
Devuelve { totalOrders, totalSpent, tier: 'bronze'|'silver'|'gold' }. Actualmente el perfil muestra "$15k" y "Gold" hardcodeados.
B-U6 🎟 Compra de entradas para eventos
POST/events/:eventId/tickets/purchase
Descuenta el precio de user.balance, genera un ticket con qrCode único (UUID firmado). Devuelve { ticket: { id, qrCode, eventId, userId, purchasedAt } }.
GET/users/me/tickets
Lista de tickets comprados. Incluye el evento asociado y el estado: valid | used | expired.
GET/users/me/tickets/:ticketId
Detalle de un ticket + QR para mostrar en pantalla o compartir.
  • Si el evento tiene maxCapacity, verificar aforo antes de permitir compra
  • El qrCode es un string único e irrepetible (ej. JWT firmado con ticketId + userId + eventId)
  • Si el usuario no tiene saldo suficiente → error 402 Insufficient Balance

Backend Endpoints a implementar / completar
B-DJ1 Dashboard — Métricas reales
GET/dj/:djProfileId/stats

B-DJ2 Cola de song requests — Gestión completa
GET/dj/:djProfileId/song-requests?status=pending
Endpoint ya existe. Añadir filtro por status.
PATCH/dj/:djProfileId/song-requests/:requestId
Body: { status: 'accepted' | 'rejected' | 'played' }. Solo accesible por el dueño del perfil DJ.
  • Socket event: song_request:new → push al DJ cuando llega una request nueva (evita polling)
  • Socket event: song_request:updated → notifica al user cuando su canción es aceptada/rechazada
B-DJ3 Gigs — CRUD completo
GET/dj/:djProfileId/gigs
Ya existe. Añadir filtro ?status=upcoming|past.
POST/dj/:djProfileId/gigs
Body: { eventId?, venueName, date, fee? }. Crear gig manual si el organizer no lo invitó.
PATCH/dj/:djProfileId/gigs/:gigId
Confirmar / cancelar gig. Body: { status: 'confirmed' | 'cancelled' }.
DELETE/dj/:djProfileId/gigs/:gigId
Eliminar gig.
B-DJ4 Promo codes — Generación en backend (seguridad)
  • Mover la generación del código al backend. Actualmente se genera en cliente con random chars → trivialmente falsificable
POST/dj/:djProfileId/promo-codes
Recibe { eventId, discountPercent, maxUses }. Genera y devuelve { code: "ARTISTA_ZYNC_XXXX", discountPercent, maxUses }.
GET/dj/:djProfileId/promo-codes/:code/stats
Cuántas veces se usó, descuento total aplicado, cuántos usos quedan.
B-DJ5 Broadcast a asistentes del evento
POST/events/:eventId/broadcast
Solo accesible para DJ con gig activo en ese evento. Envía mensaje a todos los usuarios conectados.
  • Socket event emit: event:broadcast → { djProfileId, message, type: 'announcement' | 'song' }
  • Ejemplos de uso: "Próxima canción: X", "Happy hour hasta las 2am"

Deuda Técnica Crítica

Estos puntos bloquean ir a producción con seguridad.

🔑
Spotify CLIENT_SECRET expuesto en el cliente
Las credenciales de Spotify están hardcodeadas en spotify-service.ts. Cualquiera que descompile la app puede extraerlas. Solución: crear endpoint proxy POST /music/search?q=... que llame a Spotify desde el backend.
🎲
Generación de promo codes en el cliente
El código se genera con Math.random() en el frontend. Cualquier usuario puede crear códigos de descuento sin pasar por el backend. Mover la generación completamente al servidor.
📊
Mock data en DjHomeScreen
Earnings, requests y métricas son constantes hardcodeadas ("$15.000", listas de objetos falsos). Un demo con datos reales los dejaría visibles. Reemplazar con llamadas a GET /dj/:djProfileId/stats.
💾
ZyncContext sin persistencia
Si el usuario cierra la app, pierde el establecimiento activo y debe re-escanear el QR. Persistir con AsyncStorage o derivarlo del último QR escaneado con TTL (ej. expira a las 6am).
